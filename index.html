Why raw agent access is unsafe

Without a governed runtime, an agent with database or API access might:

Mutate directly

Write to any field, bypass status transitions, ignore business rules.

Act on stale data

Approve a record that changed since the agent last read it.

Skip policy checks

Bypass role requirements, amount limits, or required approvals.

Leave no proof

No audit trail linking the decision to the policy, actor, and data version.

See it work

An AI agent approves a contract through the full safety loop. Every command is real.

Terminal 
# Scaffold and start the local mock server
$ npx create-ontologie-app my-demo --template contract-review
$ cd my-demo && dataforge dev

# Discover the model
$ dataforge schema describe --format json
{ "objectTypes": ["Contract", "Client"], "actions": ["Contract.approve", ...] }

# Find pending contracts
$ dataforge query Contract --filter-json '{"status":{"eq":"pending_review"}}' --format json
{ "data": [{ "id": "con_001", "reference": "CTR-2024-001", "amount": 45000, "status": "pending_review" }] }

# What does approve do?
$ dataforge actions describe Contract.approve --format json
{ "executionMode": "twin_apply", "riskLevel": "medium", "preconditions": ["status eq pending_review"] }

# Dry-run: create a signed plan (no mutation yet)
$ dataforge actions run Contract.approve con_001 \
    --input-json '{"comment":"Budget verified by finance"}' \
    --dry-run --format json
{ "planId": "845f9bf5-...", "hash": "sha256:7f3a...", "signature": "ed25519:...",
  "effects": [{ "before": { "status": "pending_review" }, "after": { "status": "approved" } }],
  "policyChecks": [{ "check": "precondition_status", "passed": true }, ...] }

# Inspect the plan in human-readable form
$ dataforge plan inspect 845f9bf5-... --plan-format markdown
Plan: 845f9bf5-...  |  Action: Contract.approve  |  Risk: medium
| Field     | Before           | After      |
| status    | pending_review   | approved   |
| approvedAt| null             | 2026-05-13 |

# Apply the verified plan
$ dataforge actions run Contract.approve con_001 \
    --apply-plan 845f9bf5-... \
    --plan-hash sha256:7f3a... \
    --idempotency-key approve-con-001-001 \
    --format json
{ "ok": true, "data": { "status": "approved", "version": 3, "approvedAt": "2026-05-13T..." } }

Define your model in TypeScript

A schema DSL that compiles to a versioned manifest. Fields protected by mutableBy cannot be changed through raw writes.

dataforge.schema.ts
const ContractStatus = enumType('ContractStatus', [
  'draft', 'pending_review', 'approved', 'rejected',
]);

const Contract = objectType('Contract', {
  reference: string().required().indexed(),
  amount:    number(),
  status:    ContractStatus.default('draft')
    .mutableBy(['Contract.approve', 'Contract.reject']),
  approvedAt: date().optional()
    .mutableBy(['Contract.approve']),
});

const approveContract = action('approve')
  .on(Contract)
  .executionMode('twin_apply')
  .riskLevel('medium')
  .requires(role('manager'))
  .when(c => c.status.eq('pending_review'))
  .set({ status: 'approved', approvedAt: now() });

Typed model

Contract is a declared ObjectType with constrained status values, not a free-form row.

Write protection

mutableBy locks fields to specific actions. The server rejects unauthorized writes with WRITE_POLICY_VIOLATION.

Bounded action

Contract.approve declares preconditions, effects, execution mode, and risk level. The runtime enforces all of them.

Signed plan

A dry-run produces a plan artifact with the exact diff, Ed25519 signature, policy checks, and expiry. Apply requires the same hash.

Safety is not a feature. It is the architecture.

No raw database access

Agents discover a typed schema. They never see connection strings, SQL, or internal APIs.

Scoped API keys

Server-side only. HMAC-SHA256 hashed. IP allowlist. Rate limited. Never in browser code.

Dry-run before mutation

Every change is proposed as a signed plan before anything is written.

Server-side policy

Per-field write control (mutableBy), role checks, preconditions, and 27 PlanGuard checks.

Idempotency keys

Every apply requires a unique key. Replays return the cached result without re-mutation.

Append-only audit

Every action records the plan, actor, principal type, request ID, and policy version.
